My Canonical URIs

This morning (it’s 04:41 GMT now) I made a major change to my canonical URIs (again). But this time they’re almost perfect. I believe they are now as good as they will get for a good while. The only thing likely to change with my URIs any time soon is the domain (which I expect to keep for life—which is why I’m not rushing out to buy it).

Firstly the path of the homepage of my blog was /sunc.cgi.

Then it was /sunc, which is a whole lot better.

And now it’s simply / (the above two correctly redirect to this one—even if it was /sunc.cgi/2005/01/…).

This will allow me to change to a static archive in the future without altering my canonical URIs.

A canonical URI being, simply, the URI that you want to show to people. Consider /about . That is the canonical URI for my about page. At the moment the Web server will return to you /about.html . I could very well become infatuated with PHP in the next year and decide I want the about page to be generated by /about.php. If /about.html was the canonical path then I have a problem—what about all the people with /about.html in their bookmarks or links (not to mention the search engines)? They’ll get a Document Moved Permanently (I’d put up a redirect—some Web ‘masters’ would leave it and expect people to find their own way there). Now this isn’t so bad, or is it? Say I have a few more things I want to change from HTML to PHP—my .htaccess file will soon become full of redirects (the .htaccess file for South Square Centre is full of them and it should probably have more—but alas).

My .htaccess file will start to collect Redirect directives and it will be a nightmare to keep up-to-date. And this is why I like /about. How do you do this, you ask?

Well …

Add Options +MultiViews to your .htaccess file. And that’s it. This is, of course, assuming that your server is running Apache and you’re permitted to change that particular option. It’s generally neater if you put it in the global Apache configuration if possible.

Here’s a nice introduction to good URIs by Tim Burners-Lee: Cool URIs don’t change

If you’re not already familiar, mod_rewrite is rather useful for creating ‘cool’ URIs to cover up ugly URIs. It’s not easy to learn (it took me a couple of days to get it working) but it’s a powerful tool.

Blimey, it’s 05:46 now. Good night.
